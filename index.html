Abstract

This document defines an error reporting framework which allows web developers to specify what kinds of error reports they’re interested in receiving, and the endpoints to which those reports ought to be delivered.

2. Reporting Errors

2.1. Error Reports

An error report is a collection of data about a particular error which the user agent is expected to deliver to an endpoint.

Each error report has a body, which is either `null` or a JSON object.

Each error report has an origin, which is an origin.

Each error report has an endpoint, which is either `null` or a URL.

Each error report has a kind, which is a non-empty string specifying the kind of error the report concerns.

Each error report has a timestamp, which records the time at which the report was generated, in milliseconds since the unix epoch.

2.2. The `Report-Errors` HTTP Response Header Field

The Report-Errors HTTP response header field sends a signal to the user agent that error reports of various types ought to be batched up and delivered to a particular endpoint. The header is represented by the following ABNF grammar [RFC5234]:

Report-Errors = absolute-URI *( OWS ";" [ OWS parameter ] )
parameter     = key [ BWS "=" BWS value ]
key           = token
value         = token / quoted-string

; The #rule, OWS, BWS, token, and quoted-string
; productions are all part of RFC 7230. absolute-uri is defined in RFC 3986.

Process this header, set some things on the global object. A reporting endpoint and reporting types for instance.

Periodically, the user agent SHOULD call §3.2 Deliver reports to walk through the error report store and deliver reports to the specified endpoint.

2.3. Queueing

A user agent which implements this specification will contain a error report store, which is an opaque storage mechanism that can be used to collect and persist error reports for eventual delivery. The store MUST implement the following methods, but the implementation is not exposed to the web, and is vendor-specific:

  1. Store an error report for an origin.

  2. Retrieve all stored error reports for an origin.

  3. Remove a set of error reports for an origin.

3. Algorithms

3.1. Queue error report for settings object

Given an error report (report) and an environment settings object (settings), this algorithm stores the report in the user agent’s error report store for future delivery:

  1. Let origin be the origin of settings.

  2. Set report’s origin to origin.

  3. Set report’s endpoint to the reporting endpoint of settings’s global object.

  4. Store report in the error report store for origin.

4. Privacy Concerns

4.1. Clearing the error report store

A user agent’s error report store contains data about a user’s activity on the web, and user agents ought to handle this data carefully. In particular, if a user agent gives users the ability to clear their site data or browsing history, the user agent MUST also clear the error report store.

Conformance

Conformance requirements are expressed with a combination of descriptive assertions and RFC 2119 terminology. The key words “MUST”, “MUST NOT”, “REQUIRED”, “SHALL”, “SHALL NOT”, “SHOULD”, “SHOULD NOT”, “RECOMMENDED”, “MAY”, and “OPTIONAL” in the normative parts of this document are to be interpreted as described in RFC 2119. However, for readability, these words do not appear in all uppercase letters in this specification.

All of the text of this specification is normative except sections explicitly marked as non-normative, examples, and notes. [RFC2119]

Examples in this specification are introduced with the words “for example” or are set apart from the normative text with class="example", like this:

This is an example of an informative example.

Informative notes begin with the word “Note” and are set apart from the normative text with class="note", like this:

Note, this is an informative note.
